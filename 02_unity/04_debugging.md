# 04. デバッグ（Debug.Log・Visual Studio アタッチ）

---

## 1. デバッグの概要

**デバッグ**とは「プログラムの不具合を発見・修正する作業」のことです。
Unity でのデバッグ手段は大きく2つあります。

1. **Debug.Log** を使ってログを出力して確認する方法
2. **Visual Studio のデバッガー**をアタッチして、コードの実行を一時停止して調べる方法

**どちらを使うかの判断基準：**

| 状況 | 推奨手段 |
|---|---|
| 値が正しいか確認したい | Debug.Log |
| 処理が呼ばれているか確認したい | Debug.Log |
| 複雑な条件分岐を追いたい | デバッガー |
| ループ内の特定回だけ止めたい | 条件付きブレークポイント |
| 変数がいつ・どこで変わるか追いたい | デバッガー（ウォッチ式） |

まず Debug.Log で試し、それで調べきれない場合にデバッガーを使う、
という流れが実務では一般的です。

## 2. Debug.Log の使い方

### 2-1. 基本的な出力

```csharp
using UnityEngine;

public class DebugExample : MonoBehaviour
{
    void Start()
    {
        Debug.Log("通常のログ（白）");          // 情報の確認
        Debug.LogWarning("警告ログ（黄）");      // 注意が必要な状況
        Debug.LogError("エラーログ（赤）");      // 問題が発生している状況
    }
}
```

**Console ウィンドウでの確認方法：**

1. Unity エディタのメニューから `Window → General → Console` を開きます
2. ゲームを実行すると `Debug.Log` の出力がここに表示されます
3. ログをクリックすると、対応するスクリプトの行に自動でジャンプします
4. Console ウィンドウ上部の「Clear」ボタンでログを一括削除できます

### 2-2. 変数の値を出力する

```csharp
int    hp       = 75;
string name     = "勇者";
float  posX     = transform.position.x;

// 文字列補間を使うと読みやすい
Debug.Log($"HP: {hp}");
Debug.Log($"{name} の位置: {transform.position}");

// Vector3 や Color などは自動的に文字列に変換される
Debug.Log("現在の位置: " + transform.position);
Debug.Log("子オブジェクト数: " + transform.childCount);
```

### 2-3. Debug.Log の注意点

`Debug.Log` はビルド後の実機でも実行され、パフォーマンスに影響します。
リリース前には削除または無効化が必要です。

大量の `Debug.Log` を残したままにしたくない場合は、
`[Conditional]` 属性を使って **Unity エディタ上でのみ有効**にする方法があります。

```csharp
using System.Diagnostics;
using UnityEngine;

public class DebugHelper
{
    // UNITY_EDITOR が定義されている（エディタ上の実行）場合のみ呼ばれる
    // ビルド後のプレイヤーでは呼び出し自体が除去される
    [Conditional("UNITY_EDITOR")]
    public static void Log(string message)
    {
        UnityEngine.Debug.Log(message);
    }
}
```

```csharp
// 使い方：DebugHelper.Log を使うとエディタ外では実行されない
DebugHelper.Log($"HP: {hp}");
```

## 3. Unity と Visual Studio のアタッチ手順

デバッガーを使うには Visual Studio を Unity にアタッチする必要があります。

### 3-1. 前提確認

- Visual Studio インストール時に **「Unity によるゲーム開発」ワークロード**が
  インストールされている必要があります
  （環境構築の詳細は [00_setup/environment.md](../00_setup/environment.md) を参照してください）
- Unity の外部スクリプトエディタが Visual Studio に設定されている必要があります
  `Edit → Preferences → External Tools → External Script Editor` で確認できます

### 3-2. アタッチ手順

1. Unity エディタの **Play ボタン（▶）** を押してゲームを実行します
2. Visual Studio を開きます（スクリプトをダブルクリックすると自動で開きます）
3. Visual Studio のツールバーにある **「Unity にアタッチ」ボタン** をクリックします
   または `Debug メニュー →「Unity デバッガーのアタッチ」` を選択します
4. アタッチ先の選択ダイアログで Unity のプロセスを選択して「OK」をクリックします
5. Visual Studio 下部のステータスバーが **「Unity に接続済み」** と表示されれば完了です

**よくあるハマりどころ：**

- 「Unity にアタッチ」ボタンが表示されない場合：
  Visual Studio Installer を開き「Unity によるゲーム開発」ワークロードが
  インストールされているか確認してください
- Unity 側が**実行中**（Play モード）でないとアタッチできません
  先に Unity でゲームを実行してからアタッチしてください

## 4. ブレークポイントの設置・解除

### 4-1. 設置方法

1. Visual Studio でスクリプトを開きます
2. 止めたい行の**行番号の左側の余白**をクリックします
3. **赤い丸（●）** が表示されれば設置完了です
4. Unity が実行中にその行に処理が到達すると、Visual Studio に制御が移り一時停止します

### 4-2. 解除方法

- 赤い丸を再度クリックすると解除できます
- すべてのブレークポイントをまとめて管理する場合は
  `デバッグ → ウィンドウ → ブレークポイント` で一覧を開きます

### 4-3. ブレークポイントで止まっている間にできること

- **変数にホバー**：変数名の上にマウスカーソルを乗せると現在の値がポップアップ表示されます
- **ローカルウィンドウ**：現在のスコープの変数が一覧表示されます（`デバッグ → ウィンドウ → ローカル`）
- **ウォッチウィンドウ**：任意の変数や式を登録して値を監視できます（`デバッグ → ウィンドウ → ウォッチ → ウォッチ 1`）

## 5. ステップ実行

ブレークポイントで一時停止した後、以下のキーで処理を1行ずつ追えます。

| キー | 動作 |
|---|---|
| **F5** | 続行（次のブレークポイントまで処理を再開） |
| **F10** | ステップオーバー（現在の行を実行して次の行へ。メソッドの中には入らない） |
| **F11** | ステップイン（メソッドの呼び出し先の中に入って1行ずつ追う） |
| **Shift+F11** | ステップアウト（現在のメソッドの残りを実行して呼び出し元に戻る） |

**使い分けのコツ：**
- 大まかな流れを追いたい場合 → **F10** を使う
- あるメソッドの中で何が起きているか調べたい場合 → **F11** でそのメソッドに入る
- 調べ終わったらすぐ呼び出し元に戻りたい場合 → **Shift+F11**

**使い方の例：TakeDamage() で HP が正しく減らない場合**

```csharp
public void TakeDamage(int damage)
{
    // ← ここにブレークポイントを設置して止める
    int newHp = hp - damage;    // F10 で1行ずつ進めて newHp の値を確認する
    if (newHp < 0) newHp = 0;  // F10 で進める。このガードが意図通りか確認する
    hp = newHp;                 // F10 で進めて hp に正しく代入されるか確認する
}
```

1. `TakeDamage()` の先頭行にブレークポイントを設置します
2. Unity を実行してダメージが発生する状況を作ります
3. Visual Studio に制御が移ったら **F10** で1行ずつ進めます
4. `newHp` の値をホバーで確認し、期待どおりかチェックします
5. 問題の行が特定できたら修正します

## 6. ウォッチ式・変数の確認

### 6-1. ローカルウィンドウ

ブレークポイントで止まっているとき、現在のスコープの変数が自動で一覧表示されます。

- 開き方：`デバッグ → ウィンドウ → ローカル`
- 配列や List の場合は展開して中身を確認できます

### 6-2. ウォッチウィンドウ

任意の変数や式を登録して値を継続的に監視できます。

- 開き方：`デバッグ → ウィンドウ → ウォッチ → ウォッチ 1`
- 「名前」欄に変数名や式を直接入力します

```
enemies.Count           // リストの要素数を監視する
transform.position.x    // X座標だけを確認する
hp * 2                  // 式の評価結果を確認する
hp <= 0                 // 条件式の true/false を確認する
```

### 6-3. イミディエイトウィンドウ

ブレークポイントで止まっている状態で、その場でコードを実行して確認できます。

- 開き方：`デバッグ → ウィンドウ → イミディエイト`
- 変数の値を確認したり、その場で値を変えて挙動を試したりできます

```
? hp                    // hp の値を表示する（? は「評価して表示」の意味）
hp = 100                // hp の値をその場で書き換える
? transform.position    // Vector3 の値を確認する
```

## 7. 条件付きブレークポイント

通常のブレークポイントは処理が通るたびに必ず止まりますが、
**条件付きブレークポイント**は指定した条件を満たしたときだけ止まります。

**設置方法：**

1. ブレークポイントの赤い丸を右クリックします
2. 「条件」をクリックします
3. 条件式を入力します（例：`hp <= 0`）
4. 「条件式が true の場合」が選択されていることを確認して「閉じる」をクリックします

**ヒットカウント（何回目に止まるか）の設定：**

1. ブレークポイントを右クリック →「ヒット カウント」をクリックします
2. 「ブレークポイントのヒット カウントが次の値に等しい場合に中断」などを選択して
   回数を入力します

🎮 Unity での使用例：100体の敵をループ処理するとき、特定の敵だけで止める

```csharp
for (int i = 0; i < enemyList.Count; i++)
{
    enemyList[i].TakeDamage(10); // ← ここに条件付きブレークポイントを設置
                                  //   条件：i == 50（50体目のときだけ止まる）
}
```

条件式 `i == 50` を設定すると、50体目の処理のときだけ Visual Studio が一時停止します。
すべての周回で止まる必要がないため、デバッグ効率が上がります。

## 8. よくあるハマりどころ

### アタッチしたがブレークポイントで止まらない

以下を順番に確認してください：

1. Unity が**実行中**（Play モード）になっているか
2. Visual Studio のステータスバーが「Unity に接続済み」になっているか
3. ブレークポイントが**正しいファイル**の正しい行に設置されているか
4. 該当のコードが本当に実行されているか（`Debug.Log` で通過確認する）

### ブレークポイントが中空の丸（○）になる

スクリプトにコンパイルエラーがあるか、ビルドが完了していない状態です。

1. Unity の Console ウィンドウにエラーが出ていないか確認します
2. エラーを修正して Unity がコンパイルを完了するまで待ちます
3. コンパイルが完了すると丸が赤（●）に戻ります

### Unity が実行中にスクリプトを編集するとデバッガーが切断される

Unity はスクリプトを保存するたびに再コンパイルします。
Play モード中に Visual Studio でスクリプトを保存すると
デバッガーの接続が切れることがあります。
編集が必要な場合は一度 Unity を停止してから行うのが安全です。

### Visual Studio と VSCode を混在させるとアタッチ機能が使えない

VSCode では「Unity にアタッチ」機能が Visual Studio と同じようには動きません。
デバッガーのアタッチ機能を使う場合は Visual Studio を使ってください。
