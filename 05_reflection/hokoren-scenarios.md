# 報連相ロールプレイ シナリオ集

## このファイルの使い方

このファイルは、エンジニアの実務でよく遭遇する場面を題材にした**報連相ロールプレイの練習用シナリオ集**です。

### 練習方法

**2人1組で行う場合**

1人が「受講者役」、もう1人が「先輩エンジニア役」を担当してロールプレイします。
役割を交代しながら複数のシナリオを試してみてください。

**1人で行う場合**

「自分ならどう伝えるか」を文章で書いて練習できます。
このファイル末尾の「練習メモ」フォーマットを使ってください。

### 正解は1つではありません

同じ状況でも伝え方はさまざまあります。
「この言い方が唯一の正解」ということはなく、「状況が相手に正確に伝わったか」「何をしてほしいかが明確だったか」を意識することが重要です。

### 報連相の基本的な考え方

| 種類 | 目的 |
|---|---|
| **報告** | 決まったこと・完了したことを伝える |
| **連絡** | 関係者に情報を共有する |
| **相談** | 判断が必要なことを持ちかける |

---

## ロールプレイの進め方

1. シナリオを読んで状況を把握します。
2. 受講者役が「何を・いつ・どのように伝えるか」を考えます。
3. ロールプレイを実施します（目安：2〜3分）。
4. 先輩エンジニア役からフィードバックをもらいます。
5. 「良かった点」「改善できる点」を振り返ります。

**フィードバックの観点**

- 状況が具体的に伝わったか
- 何をしてほしいかが明確だったか
- タイミングは適切だったか
- 簡潔にまとめられていたか

## シナリオ一覧

### シナリオ 1：バグを発見した

**状況**

スペースシューターの実装中に、敵が画面端で折り返す処理が特定の解像度でうまく動かないことに気づきました。
原因はわかっておらず、自分で調べても 30 分以上解決できていません。
本日中に実装を完了する予定でした。

**受講者役が知っていること**
- バグの内容と発生条件（どの解像度で・どう動かないか）
- これまでに試したこと
- 残り時間

**先輩エンジニア役の立場**
- 進捗を把握したい。
- 詰まっているなら早めに知りたい。

**考えてみましょう**
- 何を伝えますか？
- どのタイミングで伝えますか？（今すぐ・もう少し調べてから・終業前）
- どの手段（口頭・チャットツール等）で伝えますか？

**ポイント**
- 「詰まっている」と早めに伝えることで、相手が早くサポートできます。長時間1人で抱え込む必要はありません。
- 「何を試したか」を伝えることで、相手が的確なアドバイスをしやすくなります。
- 解決策がわからなくても相談して構いません。「まだ調べ中ですが…」と前置きするだけで伝わります。

---

### シナリオ 2：スケジュールが遅延しそう

**状況**

設計書の作成に想定より時間がかかっており、本日中に完成する予定でしたが、あと 2〜3 時間かかりそうな状況です。
明日の午前中に先輩エンジニアへの設計レビューが予定されています。

**受講者役が知っていること**
- 現在の進捗（どこまで完成しているか）
- 残っている作業
- 遅延の原因
- 完成の見込み時間

**先輩エンジニア役の立場**
- 明日のレビューに向けて準備したい。
- 遅延するなら早めにスケジュールを調整したい。

**考えてみましょう**
- 何を伝えますか？
- いつ伝えますか？（今すぐ・明日の朝・レビュー直前）
- レビューのスケジュールについてどう提案しますか？

**ポイント**
- 遅延がわかった時点で早めに報告することが大切です。直前の報告では相手の調整が難しくなります。
- 「遅れます」だけでなく「いつ完成するか」「何が必要か」をセットで伝えると相手が動きやすくなります。
- スケジュール変更の提案を自分から行う姿勢が、円滑なコミュニケーションにつながります。

---

### シナリオ 3：仕様の解釈が人によって異なることに気づいた

**状況**

スペースシューターの実装中に、「敵が画面下部に到達したらゲームオーバー」という仕様について、
「全ての敵が到達したらゲームオーバー」なのか「1体でも到達したらゲームオーバー」なのか、README.md を読んでも判断できませんでした。
同じチームの別の受講者に聞いたところ、違う解釈をしていることがわかりました。

**受講者役が知っていること**
- 仕様の曖昧な箇所
- 2通りの解釈の内容
- 自分はどちらだと思うか

**先輩エンジニア役の立場**
- 仕様の意図を決めた立場。
- 曖昧な箇所があれば明確化したい。

**考えてみましょう**
- 仕様の不明点はどのように確認しますか？
- 自分の解釈を伝えた上で確認する方法を考えてみましょう。
- チーム全体への共有はどうしますか？

**ポイント**
- 曖昧な仕様は「自分で判断して実装する」より確認することが重要です。後から変更が必要になるとコストが増えます。
- 「〇〇という解釈で合っていますか？」と自分の解釈を添えることで、会話がスムーズになります。
- 確認した結果はチーム全体に共有することで、同じ混乱を防げます。

### シナリオ 4：作業が完了した

**状況**

スペースシューターの実装が完了しました。
動作確認も終えており、仕様通りに動作することを確認済みです。
先輩エンジニアにコードのレビューをお願いしたい状況です。

**受講者役が知っていること**
- 完了した作業の内容
- 動作確認した内容（何をどう確認したか）
- 特にレビューしてほしい箇所

**先輩エンジニア役の立場**
- レビューの準備をしたい。
- どこに注目してレビューすればよいか知りたい。

**考えてみましょう**
- 完了報告に含めるべき内容は何ですか？
- レビュー依頼をするときに伝えておくと親切な情報は何ですか？
- 口頭とチャットツールのどちらで伝えますか？

**ポイント**
- 完了報告は「終わりました」だけでなく「何を・どこまで確認したか」を含めると相手が安心できます。
- レビュー依頼では「特に見てほしい箇所」を伝えることで、レビューの質が上がりやすくなります。
- レビューをお願いする相手の状況（忙しいか、今すぐ対応できるか）も意識すると丁寧です。

### シナリオ 5：自分のミスで他の人に影響が出た

**状況**

共有リポジトリに誤ったコードを push してしまい、他の受講者のビルドが通らなくなりました。
気づいてすぐに revert しましたが、30 分ほど影響が出てしまいました。
影響を受けた受講者と先輩エンジニアへの報告・謝罪をしなければなりません。

**受講者役が知っていること**
- 何をしてしまったか
- 影響の内容と時間
- すでに対処したこと（revert済み）
- 再発防止策のアイデア

**先輩エンジニア役の立場**
- 何が起きたかを把握したい。
- 再発防止策を一緒に考えたい。

**考えてみましょう**
- 誰に・何を・どの順番で報告しますか？
- 謝罪と説明をどのようにバランスよく伝えますか？
- 再発防止策をどう提案しますか？

**ポイント**
- ミスの報告は迅速に行うことが重要です。影響が広がる前に知らせることで対処が早くなります。
- 「謝罪」「経緯」「すでに対処した内容」「再発防止策」をセットで伝えると、相手が状況を整理しやすくなります。
- ミスを隠さず報告することがチームの信頼につながります。誰でも間違えます。大切なのはその後の対応です。

### シナリオ 6：技術的な判断を相談したい

**状況**

弾の管理に Object Pool を使うか、Instantiate/Destroy のままにするかで迷っています。
Object Pool の方がパフォーマンスは良いですが、実装に時間がかかります。
残り時間は 2 時間で、どちらを選ぶか自分では判断できない状況です。

**受講者役が知っていること**
- 2つの選択肢の内容
- それぞれのメリット・デメリット
- 残り時間
- 自分の現時点の意見（「どちらかといえば〇〇かと思っているが…」）

**先輩エンジニア役の立場**
- 技術的な判断をサポートしたい。
- 状況を把握した上でアドバイスしたい。

**考えてみましょう**
- 「どちらがよいですか？」だけでなく、何を伝えると相談しやすくなりますか？
- 自分の意見を持った上で相談することの大切さを考えてみましょう。

**ポイント**
- 相談は「判断を丸投げする」のではなく「自分の考えを持って判断を求める」ことです。
- 選択肢・メリット・デメリット・自分の意見をセットで伝えると、相手がアドバイスしやすくなります。
- 技術的な判断に唯一の正解はなく、状況（残り時間・優先度・チームの方針）によって変わります。
